# Architecture

## Stack
Requested stack: `vue-nest` (Vue 3 + NestJS, fixed by the platform at app creation).

This project was **newly scaffolded** from `template-vue-nest` in the scaffold-templates
repo — the project directory contained no prior application source (only `Dockerfile`,
`README.md`, `.github/`, `k8s/`, `kustomization.yaml`), so the full template was copied
into the project root.

## Layout
- `web/` — Vue 3 + Vite + TypeScript frontend.
  - `web/src/App.vue` — shell root; keep `data-testid="app-ready"` on the root element,
    the deploy render-gate waits on it.
  - `web/src/pages/` — route views (`Home.vue`, `About.vue`).
  - `web/src/router.ts` — Vue Router config.
  - `web/src/lib/api.ts` — fetch wrapper for calling the backend.
  - `web/vite.config.ts` — build config; `outDir` feeds the backend's served assets.
- `backend/` — NestJS backend.
  - `backend/src/main.ts` — bootstrap.
  - `backend/src/app.module.ts` — root module (Config, Prisma, tRPC, Health).
  - `backend/src/trpc/` — tRPC router wiring (`nestjs-trpc`).
  - `backend/src/users/` — example tRPC feature module (service + router).
  - `backend/src/health/` — health check controller/module.
  - `backend/src/prisma/` — Prisma client module/service.
  - `backend/prisma/schema.prisma` — Prisma schema (PostgreSQL).
  - `backend/prisma/seed.ts` — seed script.
  - `backend/Dockerfile` — backend container build (template default).
  - `web/Dockerfile.frontend` — frontend container build (template default).
- `Dockerfile` (project root) — currently the pre-existing nginx placeholder from repo
  creation. Per the technical plan, this needs to be replaced with a multi-stage build
  (Vue build → Nest build → Node runtime) that serves both API and SPA from a single
  container on port 80, matching `k8s/service.yaml`'s single `targetPort: 80`. The
  scaffolder does not overwrite this — the coder/build stage owns that change.
- `k8s/`, `kustomization.yaml` — deployment manifests (pre-existing, not modified by
  the scaffolder).
- `.pipeline/surface.json` — template-provided route/component/test-id manifest
  (backend health route, `App`/`Home` components, `app-ready`/`home-title` test ids).
  Not regenerated by the scaffolder for descriptor stacks (regeneration is enterprise-
  template-only) — update it by hand as routes/components are added.
- `colossus.yaml` — build manifest read by deploy agents (framework, output dir, base
  href, ports). Regenerated by the scaffolder on every run.
- `.colossus-acceptance.json` — acceptance contract for the post-deploy render gate.
  Seeded with the template's default stub markers (`Welcome`, `backend unreachable`) as
  `reject_signatures`; `expect_text` is left empty for the coder to fill in once the
  real bookmarks UI copy is known.

## Next steps for the developer / coder agent
1. Implement the plan's features (public bookmark list/create, About page, admin login,
   validation, seeding, graceful DB-error handling) inside `web/` and `backend/` on top
   of this scaffold — the stack is fixed as Vue 3 + NestJS; adapt any plan language that
   assumed TypeORM to the template's Prisma + tRPC setup instead.
2. Copy env templates if/when added (`backend/.env.example` → `backend/.env`, etc.) —
   none exist in the template yet; wire `DATABASE_URL`, `JWT_SECRET`, `ADMIN_EMAIL`,
   `ADMIN_PASSWORD`, `PORT` from `app-secrets` at runtime instead.
3. Run `npx prisma generate` / `npx prisma migrate dev` against a local Postgres once
   the schema is finalized (not run by the scaffolder — requires a live DB).
4. Replace the root `Dockerfile` with the multi-stage build described in the plan.
5. Update `.pipeline/surface.json` and `.colossus-acceptance.json` (`expect_text`) to
   reflect the real routes, components, test ids, and front-page copy once implemented.
6. `npm install` in both `web/` and `backend/` before running `npm run dev` / `npm run
   start:dev`.
